# rustdoc-tidy — Rust doc コメントの無駄を削る

`//!`（crate/module doc）と `///`（item doc）を、**rustdoc で実際に公開される情報密度**の観点でレビューし、事実の羅列ではなく読者が必要とする最小限に絞る。コードの正しさは見ない（correctness は別の役目）。

prose 全般に共通する観点は `prose-tidy` skill に従う。ただし括弧書きの判定には rustdoc 特有の例外があり、intra-doc リンクの括弧（`（[`crate::foo`]）`）やプラットフォーム制約の括弧（Unix のみ・非 Unix は no-op 等）は残す対象になる。以下は rustdoc に特有の追加観点。

## 前提

- 対象は Rust の doc コメント（`//!` / `///`）。設計根拠を運ぶ場合に限り、隣接する平文 `//` コメントも対象にする。
- `--document-private-items` を使うプロジェクトでは private モジュールも公開ページに載るため、private/pub を問わず全 doc コメントがレビュー対象になりうる。
- **設定/データファイルのコメントも対象にする。** プロジェクトが TOML/YAML/JSON 等の設定・データファイルを持ち、その構造（スキーマ）を Rust の型・struct の rustdoc で定義していることもある（例: `serde::Deserialize` する struct の doc）。この場合、それらの設定ファイル自身にもコメントを伴うことが多い（例: このプロジェクトの `configs/<tool>/manifest.toml`）。

## チェック観点

### 1. rustdoc の自動生成と重複させない

module doc に子モジュール一覧を箇条書きで書いていないか確認する。`cargo doc`（private を含めるプロジェクトなら `--document-private-items`）でビルドし、実際にそのモジュールの `index.html` を見る（またはグレップする）。rustdoc は `mod` 宣言から「Modules」表を各モジュール自身の doc から取った説明付きで自動生成する。手書きの一覧がその表と同じ情報なら削除する。子孫の構造（1 段下の孫モジュールの内訳）など自動生成に出ない情報だけは残す価値がある。

### 2. 設定/データファイルが rustdoc の焼き増しになっていないか確認する

設定ファイル（例 `manifest.toml`）のコメントが、その構造を定義する Rust 側（struct の `///`・その struct が属するモジュールの `//!`）で既に説明済みの**一般的な仕組み**を繰り返していないか確認する。

- 手順: 設定ファイルのコメントを読む → 該当スキーマを定義するモジュール（struct 定義・その module doc）の rustdoc を実際に読む → 同じ内容が両方にあるか突き合わせる。
- 一般仕様の再掲（属性の意味・合成のルール・条件分岐の仕組み一般など）は rustdoc 側を唯一の出所とし、設定ファイル側からは削る。
- 残すのは**そのファイルだけの理由**（なぜこの設定単位はこの値・この構成を選んだか、他の設定単位との依存関係等）。これは rustdoc には書けない、その1ファイル固有の情報なので残す。
- 例: `manifest.toml` に「`preserve = true` は既存ファイルを土台として温存する」という一般説明があり、`src/core/manifest.rs` の `preserve` フィールド doc に同じ説明が既にあるなら、`manifest.toml` 側は削り「この設定は `preserve = true` にする（既存の手動追記を壊さないため）」のようなファイル固有の理由だけを残す。

## 手順

1. スコープを決める（曖昧なら聞く）: 直前の diff だけか、crate/workspace 全体か。設定/データファイルまで含めるか。
2. 対象ファイルの doc コメントを実際に読み、`prose-tidy` の観点に加えて上記の観点 1〜2 を当てる。
3. 判定に自信が持てない箇所（rustdoc の自動生成と重複するか等）は `cargo doc` でビルドし、実際のレンダリング結果を確認してから決める。
4. 設定/データファイルも対象なら、観点 2 に従い該当スキーマの rustdoc と突き合わせる。
5. 修正後: `cargo doc`（プロジェクトの利用フラグに合わせる）・`cargo test`・`cargo fmt --check`（プロジェクトの慣習に応じて）を通す。設定ファイルを直したときは、その設定を読むプログラムのテスト（あれば）も通す。

## 出力

1. 削除した箇所の一覧（ファイル・削除理由を分類。`prose-tidy` の観点か、上記の rustdoc 固有観点 1〜2 か）。
2. 残すと判断した境界線上の箇所と、その理由（残す判断も明記して次回に活かす）。
3. ビルド/テスト/fmt の結果。
