# memory-tidy — Auto memory の棚卸し

プロジェクトの **Auto memory**（`~/.claude/projects/<project-slug>/memory/`、`MEMORY.md` 索引 ＋ 1 ファイル = 1 事実の fact ファイル群）を棚卸しし、陳腐化・重複・索引ズレを解消する。

## 方針（厳守）

- **将来の有用性で残す/消すを判断する**。陳腐化＝即削除ではない。陳腐だが今後も役立つものは事実だけ更新するか恒久的な核へトリムして残す。消化済み・他メモリやコード/git で完全に代替できるものだけを削除候補にする。
- **未確認の事実を断言しない**（report-facts-only）。陳腐化の指摘は必ず実態照合してから報告する。確認前は「〜の懸念がある」と書く。
- **不可逆操作は必ず確認する**。memory ディレクトリは **git 管理外**で、削除・上書きは取り消せない。
- 安全な修正は自動適用してよいが、削除・統合（不可逆）は一覧提示しユーザー承認後にのみ実行する。

## 手順

### 1. 対象を読む

- 現在のプロジェクトの memory ディレクトリを特定する（cwd の絶対パスを `/` → `-` に置換した slug の `~/.claude/projects/<slug>/memory/`）。
- `MEMORY.md`（索引）と全 fact ファイルを読む。

### 2. 整合性チェック（内部のみ・安価）

- **索引 ↔ ファイル**: 索引に行が無いファイル（孤立）、索引にあるがファイルが無い（欠落）。
- **壊れた `[[link]]`**: 指す先の `name` が存在しないリンク。
- **frontmatter の妥当性**: `name` / `description` / `metadata.type`（user / feedback / project / reference）の有無。
- **`MEMORY.md` の容量**: 目安 200 行 / 25KB を超えていないか。
- **重複・内容のかぶり**: 同じ事実が複数ファイルに分散していないか。

### 3. 陳腐化の能動検証

各メモリの主張を実態と照合する:

- `git log` / リポジトリ状態、参照している Issue・PR を `gh issue view` 等で状態確認（CLOSED / merged か）。
- メモリが名指しするファイル・関数・フラグ・パス・バージョンの実在確認。
- 「完了済みを進行中として書いている」「解決済みの未確定事項」「実態と異なる実装詳細」を検出する。

### 4. 残す/消すの判定

検出した各メモリを将来の有用性で判定する:

- **陳腐だが有用** → 事実部分のみ更新、または恒久的な核へトリム。
- **消化済み / 冗長**（他メモリやコード/git で完全に代替可能） → 削除候補。
- 判定には必ず理由を添える。

### 5. 適用

- **自動適用（安全）**: `MEMORY.md` 索引の同期、孤立解消 / 欠落除去、壊れたリンク修正、事実の更新 / トリム。
- **承認後に実行（不可逆）**: ファイル削除、メモリの統合。実行前に「対象・理由・代替の所在」を一覧で提示する。

### 6. 索引同期

- `MEMORY.md` を実態に合わせる。1 メモリ = 1 行（`- [Title](file.md) — hook`）。メモリ本文は索引に書かない。

## 出力

1. 検出結果（メモリ別に: 状態 / 判定 / 理由）。
2. 自動適用した安全な修正の一覧。
3. 承認待ちの不可逆操作（削除・統合）の一覧と推奨。
