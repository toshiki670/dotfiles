# prose-tidy — 文書の無駄を削る

README・CONTRIBUTING・`docs/` 配下の Markdown、PR/Issue 本文などの **prose 文書**を、読者が実際に必要とする情報密度の観点でレビューし、事実の羅列ではなく読者が必要とする最小限に絞る。コードの正しさは見ない（correctness は別の役目）。

## 前提

- 対象は人手で書く prose 文書全般（README / CONTRIBUTING / `docs/**/*.md` / PR 本文 / Issue 本文等）。Rust の doc コメント（`//!` / `///`）は対象外（`rustdoc-tidy` に譲る）。
- 自動生成される文書（CHANGELOG.md 等）は対象外。
- 判定はパターンの機械的な一致ではなく、**都度その一文を読んで**「読者はこの一文から何を得るか」を考えて行う。同じ字面でも文脈次第で残す/消すが変わる。

## チェック観点

### 1. 前置きの空語を削る

「本ドキュメントでは〜について説明する」「〜について述べる」のような、主題そのものではなく**説明の仕方について前置きするだけの語**は情報量ゼロ。主語を対象そのものにして直接何であるか・何をするかを言う。

### 2. 既に別の場所が出所の情報を書き写さない

仕様書・リンク先ドキュメント・CI のエラーメッセージ・自動生成物など、**既に別の場所が正しい出所になっている情報**を、この文書にも書き写していないか確認する。書き写せば、元の情報が変わるたびにこの文書も直す箇所が増える。

- 手順: その情報が指す出所(リンク先の仕様・CI・他セクション)を実際に開き、同じ内容が既にあるか確認する。あれば、この文書側からは削り、出所へのリンクだけを残す。
- 例: PR title に許可される型の一覧を CONTRIBUTING.md に書き出す。型の一覧は Conventional Commits の仕様と、それを検証する CI スクリプトの2箇所に既にある。3箇所目の複製は型を1つ増減するたびに更新漏れの元になる。

### 3. 読者に不要な実装事情を混ぜない

その文書の読者が意思決定・作業に使わない、**なぜこう実装したかの内部事情**(採用している具体的な機構名、内部フローの詳細等)を書いていないか確認する。読者が必要とするのは結果としての「ルール」であって、そのルールを支える「実装の都合」ではない。実装は変わりうるが、実装事情を理由に書いてしまうとルールの文言もその都度書き直す羽目になる。

- 例: 「PR は squash merge で main に入るため、PR title がそのまま commit message になる。だから PR title を Conventional Commits 形式にする」。読者が従うべきルールは「PR title を Conventional Commits 形式にする」だけで足りる。squash merge かどうかはマージ方式の運用次第でいつ変わってもおかしくなく、読者の行動も変えない。

### 4. 意味の薄い括弧書きを削る

`（…）` が直前の文をそのまま言い換えているだけなら削除する。判定基準は「この括弧を消したら、読者は失う事実があるか」。型・スコープを特定する括弧、非自明な理由を説明する括弧、具体例を列挙する括弧は残す。直前の文の言い換えでしかない括弧は削る。

### 5. 用語は使う前に定義する

ドメイン固有の語彙が、定義される前の文で使われていないか確認する。文書の冒頭・上位セクションから個別セクションへ、「既知から未知へ」語彙を積み上げる。まだ登場していない語をいきなり使わない。

### 6. この文書を書く際に使った内部の抽象語彙を持ち込まない

社内的な設計方針の議論で使う抽象語彙(例: SSOT・ライフサイクル・安定/揮発等)を、その語彙を前提としない読者向けの文書にそのまま書いていないか確認する。具体的な技術的事実に置き換える。

## 手順

1. スコープを決める(曖昧なら聞く): 対象ファイル1つか、diff全体か。
2. 対象文書を実際に読み、観点1〜6を順に当てる。パターンで機械的に判定せず、周辺の文脈込みで都度読む。
3. 判定に自信が持てない箇所(SSOTの出所か等)は、実際にリンク先・CI・他セクションを開いて確認してから決める。
4. 少数のファイルは直接編集する。10ファイルを超えるような機械的な一括修正は、判定基準と before/after を明記したうえでサブエージェントへ委任してよい。

## 出力

1. 削除した箇所の一覧(ファイル・削除理由を観点1〜6のどれかで分類)。
2. 残すと判断した境界線上の箇所と、その理由(残す判断も明記して次回に活かす)。
