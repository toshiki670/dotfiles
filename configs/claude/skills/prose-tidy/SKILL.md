# prose-tidy — 文書の無駄を削る

人間の読者に向けて書く **prose 文書**を、読者が実際に必要とする情報密度の観点でレビューし、事実の羅列ではなく必要最小限に絞る。内容の正しさ（事実関係）は見ない（correctness は別の役目）。

## 前提

- 対象はファイル種別で決めない。**人間が読み、正誤ではなく情報密度・構成が問題になる text か**で判断する。Markdown 文書(README / CONTRIBUTING / `docs/**/*.md` 等)・PR/Issue 本文・レビューコメント・commit message の body・CLI の help/エラーメッセージ・(Rust 以外の言語の)コード内平文コメント等が該当しうる。Rust の doc コメント（`//!` / `///`）とその隣接コメントは対象外（`rustdoc-tidy` に譲る）。
- 自動生成される文書（CHANGELOG.md 等）は対象外。
- model 向けの設定文書(CLAUDE.md・skill 定義・memory ファイル等、人間読者を前提としない文書)には観点5(用語は使う前に定義する)・観点6(抽象語彙を持ち込まない)は当てはまらない。
- 判定はパターンの機械的な一致ではなく、**都度その一文を読んで**「読者はこの一文から何を得るか」を考えて行う。同じ字面でも文脈次第で残す/消すが変わる。

## チェック観点

### 1. 前置きの空語を削る

「本ドキュメントでは〜について説明する」「〜について述べる」のような、主題そのものではなく**説明の仕方について前置きするだけの語**は情報量ゼロ。主語を対象そのものにして直接何であるか・何をするかを言う。

### 2. 既に別の場所が出所の情報を書き写さない

仕様書・リンク先ドキュメント・CI のエラーメッセージ・自動生成物など、**既に別の場所が正しい出所になっている情報**を、この文書にも書き写していないか確認する。書き写せば、元の情報が変わるたびにこの文書も直す箇所が増える。

- 手順: その情報が指す出所（リンク先の仕様・CI・他セクション）を実際に開き、同じ内容が既にあるか確認する。あれば、この文書側からは削り、出所へのリンクだけを残す。
- 例: PR title に許可される型の一覧を CONTRIBUTING.md に書き出す。型の一覧は Conventional Commits の仕様と、それを検証する CI スクリプトの2箇所に既にある。3箇所目の複製は型を1つ増減するたびに更新漏れの元になる。

### 3. 経緯・実装事情を残すかどうかは文書の目的で決める

「なぜこう実装したか」という経緯・内部事情を書くべきかどうかは、文書のジャンルによって答えが逆転する。一律に禁止するのではなく、次の手順で判定する。

1. 対象文書の目的(ジャンル)を特定する。その文書の存在意義自体が「決定の経緯を記録すること」であれば(例: 設計判断記録、commit message の body、議論そのものである PR/Issue のコメント)経緯は残す。「今の状態・今どうすべきか」を伝えることが目的の文書(使い方ガイド、ルール集等)であれば、経緯は原則削る。
2. ジャンルのデフォルトに関わらず、次のいずれかに該当する記述は残す:
   - **Chesterton's Fence 予防**: 理由を知らないと元に戻されてしまう拒否ロジック・制約
   - **分類・判断基準そのもの**: 理由が選択肢を分ける基準として機能している
3. **対象リポジトリに既存の文書規約や同ジャンルの実例があれば、この一般則より先にそちらへ従う。** リポジトリ固有のジャンル分けを定義した skill があれば(例: このリポジトリの `prose-genres`)それを使う。

- 例: 「PR は squash merge で main に入るため、PR title がそのまま commit message になる。だから PR title を Conventional Commits 形式にする」。ルール集ジャンルの文書では経緯は原則削る対象で、この理由は分類基準にも Chesterton's Fence にも該当しないため削ってよい。読者が従うべきルールは「PR title を Conventional Commits 形式にする」だけで足りる。

### 4. 意味の薄い括弧書きを削る

`（…）` が直前の文をそのまま言い換えているだけなら削除する。判定基準は「この括弧を消したら、読者は失う事実があるか」。型・スコープを特定する括弧、非自明な理由を説明する括弧、具体例を列挙する括弧は残す。

### 5. 用語は使う前に定義する

ドメイン固有の語彙が、定義される前の文で使われていないか確認する。文書の冒頭・上位セクションから個別セクションへ、「既知から未知へ」語彙を積み上げる。まだ登場していない語をいきなり使わない。

### 6. この文書を書く際に使った内部の抽象語彙を持ち込まない

社内的な設計方針の議論で使う抽象語彙（例: SSOT・ライフサイクル・安定/揮発等）を、その語彙を前提としない読者向けの文書にそのまま書いていないか確認する。具体的な技術的事実に置き換える。

## 手順

1. スコープを決める（曖昧なら聞く）: 対象ファイル1つか、diff全体か。
2. 対象文書を実際に読み、観点1〜6を順に当てる。パターンで機械的に判定せず、周辺の文脈込みで都度読む。
3. 判定に自信が持てない箇所（出所として妥当か等）は、実際にリンク先・CI・他セクションを開いて確認してから決める。
4. 少数のファイルは直接編集する。10ファイルを超えるような機械的な一括修正は、判定基準と before/after を明記したうえでサブエージェントへ委任してよい。
5. 修正後: プロジェクトに Markdown lint があれば通す（このリポジトリなら `mise run lint` の rumdl）。削除・移動した見出しやセクションへのリンク・アンカー参照が壊れていないか確認する。

## 出力

1. 削除した箇所の一覧（ファイル・削除理由を観点1〜6のどれかで分類）。
2. 残すと判断した境界線上の箇所と、その理由（残す判断も明記して次回に活かす）。
3. lint・リンク切れ確認の結果。
