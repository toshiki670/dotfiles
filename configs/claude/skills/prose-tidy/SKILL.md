# prose-tidy — 文書の無駄を削る

人間の読者に向けて書く **prose 文書**を、読者が実際に必要とする情報密度の観点でレビューし、事実の羅列ではなく必要最小限に絞る。内容の正しさ（事実関係）は見ない（correctness は別の役目）。

## 前提

- 対象はファイル種別で決めない。**人間が読み、正誤ではなく情報密度・構成が問題になる text か**で判断する。Markdown 文書(README / CONTRIBUTING / `docs/**/*.md` 等)・PR/Issue 本文・レビューコメント・commit message の body・CLI の help/エラーメッセージ・コード内平文コメント等が該当しうる。
- 特定言語の doc コメント生成ツールが解釈する専用記法(例: Rust の `//!` / `///`)は、その記法特有の構造(自動生成される要素一覧・相互参照リンク等)をこの skill の一般則で判定すると誤判定しうる。手順1でその記法を扱う専用の skill・rule が見つかれば、その範囲はそちらに譲る。
- 自動生成される文書（CHANGELOG.md 等）は対象外。
- model 向けの設定文書(CLAUDE.md・skill 定義・memory ファイル等、人間読者を前提としない文書)には観点5(用語は使う前に定義する)・観点6(抽象語彙を持ち込まない)は当てはまらない。
- **観点2・3は、対象文書が「この先も編集され続けるか」「一度書いたら書き換えないか」で判定が変わる。** README・ルール集・コード内コメントのように**この先も編集され続ける文書**は、書いた時点の事実がいずれ変わりうる。そこに他所と重複する情報や不要な経緯を持たせるほど、変更のたびに複数箇所を直す羽目になり、直し漏れは見つけにくい。一方 commit message の body・PR/Issue 本文のように**一度書いたら書き換えない記録**は、この意味での同期コストを負わない。むしろ書いた時点の文脈をその場で自己完結させる方が、後から読む人にとって価値が高い。
- **経緯に見える記述は、実装のメカニズム・時系列(How/History)と設計の理由(Why)のどちらかで置き場所が変わる。** How/History は直前の項目の軸(生きた文書・記録のどちらか)に従う。Why(この機能・設計が存在する理由)は生きた文書にこそ残す価値があり、観点3の対象外。
- 判定はパターンの機械的な一致ではなく、**都度その一文を読んで**「読者はこの一文から何を得るか」を考えて行う。同じ字面でも文脈次第で残す/消すが変わる。

## チェック観点

### 1. 前置きが空語かどうかは文書の目的で決める

「本ドキュメントでは〜について説明する」のような前置きは、**読者が既に知っている・何も予告していないときだけ**空語になる。正式な仕様書・チュートリアルなど、要約的な前置きがジャンルの慣行として期待される文書では、同じ形の文が空語ではなく構造的なシグナルとして機能する。一律に禁止するのではなく、次の手順で判定する。

1. その前置きを消したら、読者は何を失うか(主題・この先で何が得られるかの予告)を考える。何も失わないなら空語。
2. そのジャンルで前置きが慣行として期待されているか(手順1の調査で確認)。期待されているなら残す。

### 2. この先も編集され続ける文書では、既に別の場所が出所の情報を書き写さない

仕様書・リンク先ドキュメント・CI のエラーメッセージ・自動生成物など、**既に別の場所が正しい出所になっている情報**を、この先も編集され続ける文書に書き写していないか確認する(前提の判定軸を参照)。

- 手順: その情報が指す出所（リンク先の仕様・CI・他セクション）を実際に開き、同じ内容が既にあるか確認する。あれば、この文書側からは削り、出所へのリンクだけを残す。
- 例: PR title に許可される型の一覧を CONTRIBUTING.md(この先も編集され続けるルール集)に書き出す。型の一覧は Conventional Commits の仕様と、それを検証する CI スクリプトの2箇所に既にある。3箇所目の複製は型を1つ増減するたびに更新漏れの元になる。

### 3. 実装のメカニズム・経緯(How/History)を残すかどうかは対象文書が生きているか履歴かで決める(設計の理由 = Why は対象外)

「なぜこう実装したか」に見える記述は、**実装のメカニズム・時系列(How/History)** なのか、**設計の理由(Why)** なのかをまず見極める。History/How だけを、次の手順で判定する。

1. 対象文書がどちらかを特定する。**一度書いたら書き換えない記録**で、その存在意義自体が「決定の経緯を記録すること」であれば(例: 設計判断記録、commit message の body、議論そのものである PR/Issue のコメント)残す。**この先も編集され続ける文書**(使い方ガイド、ルール集等)であれば、原則削る。
2. 編集され続ける文書でも、次のいずれかに該当する記述は残す:
   - 理由を知らないと元に戻されてしまう拒否ロジック・制約(Chesterton's Fence 予防。却下した代替案の理由=Why not)
   - 理由が選択肢を分ける基準として機能している(分類・判断基準そのもの)
3. **どちらにも該当しない History/How は、たとえ理由付けのように見えても大抵はゴミ情報。** とりわけ、今の成果物には存在しない・関係しない対象への言及(「〜という方法も検討したが使わなかった」「package.json は持たない」等、開発中に試して捨てた選択肢の記録)は、ほぼ確実に該当しない。判定: **この一言は読者の疑問に答えているか、それとも書き手が開発中に通った道筋を反射的に書き残しているだけか。** 後者ならコミットメッセージ・PR概要に譲り、成果物からは削る。

- 例(Why は残す): CONTRIBUTING.md が「PR title を Conventional Commits 形式にする」理由として「release-plz がコミットタイプからバージョンバンプ・CHANGELOG を自動生成するため」を書くのは Why であり、生きた文書でも残してよい。
- 例(How の混入): 「PR は squash merge で main に入るため、PR title がそのまま commit message になる。だから PR title を Conventional Commits 形式にする」。前半は現在の実装メカニズム(How)であって Why ではない。ルール集(この先も編集され続ける文書)では原則削る対象で、この理由は分類基準・Chesterton's Fence のいずれにも該当しないため削ってよい。読者が従うべきルールは「PR title を Conventional Commits 形式にする」だけで足りる。
- 例(開発過程の残骸): あるツールを npx でその場実行するよう変えた際、コメントに「package.json は持たない」を書き足す。これは今の成果物(そのコメントが付いた行)を理解するのに不要で、読者はそもそも package.json の存在を期待していない。開発の過程で package.json を作って撤去した、という経緯はコミットメッセージに書けば足りる。

### 4. 意味の薄い括弧書きを削る

`（…）` が直前の文をそのまま言い換えているだけなら削除する。判定基準は「この括弧を消したら、読者は失う事実があるか」。型・スコープを特定する括弧、非自明な理由を説明する括弧、具体例を列挙する括弧は残す。

### 5. 用語は使う前に定義する

ドメイン固有の語彙が、定義される前の文で使われていないか確認する。文書の冒頭・上位セクションから個別セクションへ、「既知から未知へ」語彙を積み上げる。まだ登場していない語をいきなり使わない。

### 6. この文書を書く際に使った内部の抽象語彙を持ち込まない

社内的な設計方針の議論で使う抽象語彙（例: SSOT・ライフサイクル・安定/揮発等）を、その語彙を前提としない読者向けの文書にそのまま書いていないか確認する。**これらの語自体は他の文脈でも一般的に使われうる**(SSOT は他プロジェクト独自の Source of Truth を指すこともあり、安定/揮発も他の設計原則が独自に定義しうる)。語が出現しているかどうかではなく、**その用法が実際に書き手側の内部設計方針を参照しているか**で判断する。参照しているなら具体的な技術的事実に置き換える。

## 手順

1. **実行環境を調査する。**
   - Claude Code の Auto Memory(`~/.claude/projects/<slug>/memory/`、`<slug>` は cwd の絶対パスの `/` を `-` に置換したもの)が使えるなら、まず `doc-conventions.md` の有無を確認する。あれば読んで規約・傾向として使い、以下の再調査はしない。
   - 無ければ、このプロジェクトに対象文書の種別・記法に関わる skill・rule・CONTRIBUTING 等の文書規約がないか確認する。明文化された規約が無くても、同じジャンルの既存ファイルを実際に読み、そこから読み取れる傾向(トーン・構成・記法)を把握する。見つけた内容は(Auto Memory が使えるなら)`doc-conventions.md` として保存し、次回以降はそちらを参照する。内容が現状と食い違うと気づいたら、その場で実態に合わせて更新する。
   - **見つかった規約・傾向は、下の観点1〜6のどれに対しても、この skill の一般則より優先して適用する**(経緯を残すかどうかの観点3に限らない。例: 空語に見える前置きを意図的に使う文体規約、用語を定義せず使う前提のグロッサリ等)。対象文書の種別・記法に特化した skill が見つかれば、その範囲は重複して適用しない。
2. **スコープを決める（曖昧なら聞く）**: 対象ファイル1つか、指示された diff 全体か。依頼されていない範囲(プロジェクト全体等)へ自発的に広げない。
3. 対象文書を実際に読み、観点1〜6を順に当てる。
4. 判定に自信が持てない箇所（出所として妥当か等）は、実際にリンク先・CI・他セクションを開いて確認してから決める。
5. 少数のファイルは直接編集する。10ファイルを超えるような機械的な一括修正は、判定基準と before/after を明記したうえでサブエージェントへ委任してよい。
6. **修正後の検証: 手順2で決めたスコープの範囲(編集したファイルだけ)に textlint を通す。プロジェクト全体のlintコマンドをこちらから実行しない。**
   - 手順1でこのプロジェクト固有の lint の仕組みが見つかっていれば、そちらでその対象ファイルだけを検証する。
   - 見つからなければ、事前インストール不要で次を実行する:
     `npx -y -p textlint -p textlint-rule-preset-ja-technical-writing -p textlint-rule-preset-ai-writing textlint <編集したファイル>`
   - 削除・移動した見出しやセクションへのリンク・アンカー参照が壊れていないかも確認する。

## 出力

1. 削除した箇所の一覧（ファイル・削除理由を観点1〜6のどれかで分類）。
2. 残すと判断した境界線上の箇所と、その理由（残す判断も明記して次回に活かす）。
3. lint・リンク切れ確認の結果。
