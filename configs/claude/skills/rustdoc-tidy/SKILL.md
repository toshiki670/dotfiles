# rustdoc-tidy — Rust doc コメントの無駄を削る

`//!`（crate/module doc）と `///`（item doc）を、**rustdoc で実際に公開される情報密度**の観点でレビューし、事実の羅列ではなく読者が必要とする最小限に絞る。コードの正しさは見ない（correctness は別の役目）。

## 前提

- 対象は Rust の doc コメント（`//!` / `///`）。設計根拠を運ぶ場合に限り、隣接する平文 `//` コメントも対象にする。
- `--document-private-items` を使うプロジェクトでは private モジュールも公開ページに載るため、private/pub を問わず全 doc コメントがレビュー対象になりうる。
- 判定はパターンの機械的な一致ではなく、**都度そのコメントを読んで**「rustdoc の読者はこの一文から何を得るか」を考えて行う。同じ字面でも文脈次第で残す/消すが変わる。

## チェック観点

### 1. 前置きの空語を削る

「中心は」「本体は」「〜の入口」「〜のホーム」「〜を担う」のような、主題そのものではなく**説明の仕方について前置きするだけの語**は情報量ゼロ。crate/module の要約 1 文目は、主語を対象そのものにして直接何をするかを言う。

- NG: `中心は core（本体）: 設定を配置する`
- OK: `core は設定を配置する`

### 2. rustdoc の自動生成と重複させない

module doc に子モジュール一覧を箇条書きで書いていないか確認する。`cargo doc`（private を含めるプロジェクトなら `--document-private-items`）でビルドし、実際にそのモジュールの `index.html` を見る（またはグレップする）。rustdoc は `mod` 宣言から「Modules」表を各モジュール自身の doc から取った説明付きで自動生成する。手書きの一覧がその表と同じ情報なら削除する。子孫の構造（1 段下の孫モジュールの内訳）など自動生成に出ない情報だけは残す価値がある。

### 3. 旧実装との比較を消す

「旧 `X.fish` の移植」「旧 bash の `Y()` 相当」「出力は従来と不変」のような、**移行元・旧版との比較**は git 履歴・コミットメッセージが持つべき情報で、今の挙動を理解するのに不要。比較の句だけを削り、地の文の事実（今の挙動そのもの）は残す。比較が文の全内容だった行は行ごと削除する。

例外: 「旧キー X は廃止済みで、黙って無視せずロード時エラーにする」のように、**今の挙動（なぜ拒否するか）の非自明な根拠**になっている場合は残す。これは移行史の言及ではなく、意図的なバリデーションの理由（Chesterton's Fence の防止）。

### 4. 意味の薄い括弧書きを削る

`（…）` が直前の文をそのまま言い換えているだけなら削除する。判定基準は「この括弧を消したら、読者は失う事実があるか」:

- 削除すべき例: 「〜する（clap は持たない）」← 直前の「引数は取らず」の言い換え
- 残すべき例: 型・スコープを特定する括弧（`dst`（ディレクトリ）等）、非自明な理由を説明する括弧（なぜ mtime でなく内容ハッシュか等）、具体例を列挙する括弧（対象 5 本の一覧等）、プラットフォーム制約（Unix のみ・非 Unix は no-op 等）、intra-doc リンクそのもの（`（[`crate::foo`]）`）。

### 5. 用語は使う前に定義する

ドメイン固有の語彙が、定義される前のコメントで使われていないか確認する。crate doc → 主要モジュールの doc → 個別実装、の順で「既知から未知へ」語彙を積み上げる。孫モジュールが定義済みの語を前提に書くのは良いが、まだ登場していない語をいきなり使わない。

### 6. 設計方針ドキュメント固有の抽象語彙を持ち込まない

プロジェクトに内部設計方針・アーキテクチャ文書がある場合、その文書だけで通じる抽象語彙（例: SSOT・ライフサイクル・安定/揮発 等）を、定義なしにコードのコメントへそのまま持ち込まない。具体的な技術的事実に置き換える。

## 手順

1. スコープを決める（曖昧なら聞く）: 直前の diff だけか、crate/workspace 全体か。
2. 対象ファイルの doc コメントを実際に読み、観点 1〜6 を順に当てる。パターンで機械的に判定せず、周辺の文脈込みで都度読む。
3. 判定に自信が持てない箇所（rustdoc の自動生成と重複するか等）は `cargo doc` でビルドし、実際のレンダリング結果を確認してから決める。
4. 少数のファイルは直接編集する。10 ファイルを超えるような機械的な一括修正は、判定基準と before/after を明記したうえでサブエージェントへ委任してよい。
5. 修正後: `cargo doc`（プロジェクトの利用フラグに合わせる）・`cargo test`・`cargo fmt --check`（プロジェクトの慣習に応じて）を通す。

## 出力

1. 削除した箇所の一覧（ファイル・削除理由を観点 1〜6 のどれかで分類）。
2. 残すと判断した境界線上の箇所と、その理由（残す判断も明記して次回に活かす）。
3. ビルド/テスト/fmt の結果。
