# textlint-fix — textlint の指摘を意味・声・リズムを保ったまま解消する

textlint が指摘した箇所に、ルールの regex を満たすことだけを目的とした機械的な書き換えで応じない。指摘対象の一文が伝える意味・声・含意をまず特定し、それを保ったまま指摘を解消する。情報密度・媒体適合の判断は見ない（prose-tidy の役目）。内容の正しさ（事実関係）も見ない（correctness は別の役目）。

## 前提

- 対象は人間が読む prose 文書。README・CONTRIBUTING・PR/Issue 本文・レビューコメント・commit message の body 等、prose-tidy と同じ対象範囲。
- CLAUDE.md・skill 定義(SKILL.md)・memory ファイル等、Claude Code がモデルとして読む設定文書は対象外。これらには textlint（`preset-ai-writing` を含む全ルール）をそもそも実行しない。「人間読者にとって AI 生成っぽく見えるか」は、モデルが読む設定文書には本質的に当てはまらない判定のため。
- prose-tidy とは別パス。prose-tidy は密度・媒体適合を判断し、この skill は lint 指摘への対応を担う。どちらかがどちらかを自動的に呼び出すことはなく、必要になったときにそれぞれ個別に起動する。

## チェック観点

判定はルールの regex を機械的に満たすことではなく、都度その一文を読んで「元の意味・声・含意のうち何を保つべきか」を特定してから変換を選ぶ。同じルール指摘でも、文脈次第で選ぶ変換は変わる。

### 1. 文分割は複合主語・述語の対応関係を保つ

`sentence-length` 等を理由に一文を分割するとき、分割後の各文で述語がどの主語・スコープに対応するかを元文と突き合わせる。一部にしかかからない修飾がもう一方の文にも及んでいるように読めないか確認する。

- 例: 「`configs/` 配下の設定と `bin/` 配下のスクリプトが〜」を分割した結果、両方が `bin/` 配下であるかのように読める文になっていないか。

### 2. `no-ai-list-formatting` は説明文への展開よりリズムを保つ表現を優先する

`**Label**は「…」と問う` のような説明的な言い換えへ展開する前に、em-dash 形式(`- **Label** — 説明`)のような、元の定型的なリズムを保ったまま構造だけ変える手段を検討する。

### 3. `no-ai-hype-expressions` は強度・条件を弱めない同義語に置き換える

指摘対象語（「完全に」等）を避けるとき、置き換えた同義語が元の判定基準の厳しさを緩めていないか確認する。「完全に代替できる」→「だけで足りる」のような書き換えは、必須条件を十分条件へ緩めてしまっている。

### 4. 言い換え後も時系列・条件の含意が残っているか確認する

「A してから B にする」のような順序・条件の含意を持つ文を言い換えるとき、その含意（A が済んで初めて B になりうる、等）が言い換え後の文にも残っているか、元文と読み比べて確認する。

## 手順

1. 対象ファイル・スコープを決める（曖昧なら聞く）: 1ファイルか、指摘された diff 全体か。依頼されていない範囲へ自発的に広げない。決めたら、前提の除外対象（モデル向け設定文書）に該当するファイルをその時点で外す。
2. 対象プロジェクトが独自の `.textlintrc.json`（プロジェクト固有のルール上書き）を持つか確認する。無ければ、textlint 自身の config 探索（対象ファイルの祖先ディレクトリを遡る）によりグローバル設定（`~/.textlintrc.json`）がそのまま使われる。プロジェクト固有の lint 実行手段（package.json のスクリプト・CI 定義等）が見つかればそちらを使う。見つからなければ次を実行する:
   `npx -y -p textlint@15.7.1 -p textlint-rule-preset-ja-technical-writing@12.0.2 -p textlint-rule-preset-ai-writing@1.1.0 textlint --ignore-path /dev/null <対象ファイル>`
   `--ignore-path /dev/null` は、対象ファイルが ignore 設定で黙って除外され、未検証のまま成功したことにならないようにするため。
3. 指摘を1件ずつ確認する。チェック観点1〜4に該当するものはその指針に沿って変換し、該当しないものも都度その一文を読んで元の意味・声・含意を保つ最小限の変換を選ぶ。
4. 修正前後を突き合わせ、意味・声・含意が変わっていないか確認する。
5. 再度 textlint を実行し、対象の指摘が解消され新たな指摘を生んでいないか確認する。
6. ルールを満たしつつ意味を保つ変換が見つからない指摘は、無理に押し込めず「このルールとこの文がなぜ衝突するか」を出力に残す。

## 出力

1. 修正した指摘の一覧（ルール名・ファイル・変換内容）。
2. 意味・声を保てず対応を見送った指摘（あれば、衝突の理由とともに）。
3. 再実行結果。
