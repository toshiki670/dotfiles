# memory-tidy — Auto memory の棚卸し

プロジェクトの **Auto memory**（`~/.claude/projects/<project-slug>/memory/`、`MEMORY.md` 索引 ＋ 1 ファイル = 1 事実の fact ファイル群）を棚卸しし、陳腐化・重複・索引ズレを解消する。

## 方針（厳守）

- **将来の有用性で残す/消すを判断する**。陳腐化＝即削除ではない。陳腐だが今後も役立つものは事実だけ更新するか恒久的な核へトリムして残す。消化済み・他メモリやコード/git だけで足りるものだけを削除候補にする。
- **「正しいか」と「今後も参照され得るか」は別軸**。前者は3で実態照合する。後者は自動計測できない（読了ログは存在せず、読まずに内在化される知識もある）ので、frontmatter に最終利用日時のような欄は設けない。日付は本文中に明記する運用（例:「2026-07-05, PR#585 で…」）に一本化し、内容を更新したときだけ書く。**以降「更新日」は本文中に現れる最新の明記日付を指す。明記日付が無いメモリは「更新日不明」として扱い、それ自体を陳腐化の根拠にしない**（古くから触っていない安定したメモリと、単に日付を書いていないだけのメモリを区別できないため）。
- **未確認の事実を断言しない**（report-facts-only）。陳腐化の指摘は必ず実態照合してから報告する。確認前は「〜の懸念がある」と書く。MCP/外部サイトが未接続・未到達で照合できない場合も同様に「未検証」と明記し、断定しない。
- **不可逆操作は必ず確認する**。memory ディレクトリは **git 管理外**で、削除・上書きは取り消せない。
- 安全な修正は自動適用してよいが、削除・統合（不可逆）は一覧提示しユーザー承認後にのみ実行する。

## 手順

### 1. 対象を読む

- 現在のプロジェクトの memory ディレクトリを特定する（cwd の絶対パスを `/` → `-` に置換した slug の `~/.claude/projects/<slug>/memory/`）。
- `MEMORY.md`（索引）と全 fact ファイルを読む。
- **7 の階層化（カテゴリ別サブディレクトリ＋サブ索引）が既に行われていないか確認する。** 行われていれば、以降の手順（2・6 含む）は「トップレベルの `MEMORY.md` はカテゴリ単位」「各サブ索引は fact 単位」という2段構成を前提に読み替える。フラットな1段構成だと誤認してカテゴリ化を元へ戻さない。

### 2. 整合性チェック（内部のみ・安価）

- **索引 ↔ ファイル**は、索引に行が無いファイル（孤立）や索引にあるがファイルが無い（欠落）が無いか確認する。7 で階層化済みの場合はカテゴリのサブ索引を単位に見る（fact を直接トップレベル `MEMORY.md` と突き合わせて「孤立」と誤検出しない）。
- **壊れた `[[link]]`**とは、指す先の `name` が存在しないリンクを指す。
- **frontmatter の妥当性**は、`name` / `description` / `metadata.type`（user / feedback / project / reference）の有無を確認する。
- **`MEMORY.md` の読み込み上限（目安ではなく実際の切り捨てライン）**は、先頭 200 行 or 25KB（どちらか早い方）を超えた分がセッション開始時に一切読み込まれないことを指す（Claude Code 公式仕様）。超過は「見た目が長い」程度の問題ではなく、超過分のメモリが今後のセッションから実質的に消えている状態＝最優先で是正する。`wc -l` / `wc -c` で実測して確認する。
- **索引の1行あたりの長さ**は、目安 150 字以内かを確認する。索引行が本文の要約でなく全文複製になっている（＝本来ファイル側にしかないはずの詳細が索引に漏れ出している）のは、索引が「索引」の役目を外れているサインだ。
- **重複・内容のかぶり**は、同じ事実が複数ファイルに分散していないかを確認する。

### 3. 陳腐化の能動検証

各メモリの主張を実態と照合する。事実として正しいかどうかだけを見る（今後も参照され得るかは 4 で別途扱う）。

- `git log` / リポジトリ状態、参照している GitHub Issue・PR を `gh issue view` 等で状態確認（CLOSED / merged か）。
- **GitHub 以外のチケット系**（Linear / Jira / Notion 等）を参照するメモリは、対応する MCP ツールで現在の状態を確認する。
- **外部サイトの記載を根拠にしたメモリ**（バージョン・公式ドキュメントの仕様等）は WebFetch/WebSearch で再取得し突き合わせる。
- メモリが名指しするファイル・関数・フラグ・パス・バージョンの実在確認。
- 「完了済みを進行中として書いている」「解決済みの未確定事項」「実態と異なる実装詳細」を検出する。
- **MCP 未接続・未認証や到達不能で確認できない場合は「不明」として扱う。** 確認できなかったことを「陳腐化なし」の根拠にしない（report-facts-only）。

### 4. 有用性の判定（正確性 ≠ 参照され得るか）

3 で事実確認した上で、**「今後も参照され得るか」を type ごとに判定する**:

| type | 陳腐化の起点 | 判定材料 |
|---|---|---|
| `project` | 初期タスク/構想の完了 | 参照する Issue/PR が閉じ、追跡すべき未完項目も残っていなければ圧縮/削除候補 |
| `feedback` | 対象の状況・ツール自体の消滅 | 更新日の古さでは判断しない（低頻度発火が正常）。対象の状況/ツールがまだ存在するかだけを見る |
| `reference` | 対象ツール/コマンド/バージョンの引退 | 3 の実在確認（ファイル/コマンド/バージョンが実際に無くなっているか）が主判定材料 |
| `user` | ユーザーの役割/前提の変化 | 明示的な自己申告がない限り自動では削除しない |

被リンク数（`[[link]]`が0件）は `project` / `reference` の判定に使ってよいが、**単独のトリガーにしない**。この2 type でも被リンク0は普通に起きる（`feedback` と同様、単独で効く孤立したメモリは健全にありうる）ため、3 で陳腐化が実際に判明した場合の**補強材料**としてのみ使う。`feedback` はそもそも対象外（強いルールほど他から参照されず単独で効くのが正常）。

事実確認は通ったが、上記の弱いシグナル（project 完了・reference 孤立）が重なる中間層は削除候補とせず、まずユーザーへ一言確認する（例:「最近効いた場面はありますか」）。**更新日の古さ単体をトリガーにしない**（`feedback` を毎回引っかけてしまう）。

各メモリの最終判定には必ず理由を添える。

- **陳腐だが有用** → 事実部分のみ更新、または恒久的な核へトリム。
- **消化済み / 冗長**（他メモリやコード/git だけで足りる） → 削除候補。
- **判断つかず** → ユーザーに確認。

### 5. 適用

- **自動適用（安全）**は、`MEMORY.md` 索引の同期、孤立解消 / 欠落除去、壊れたリンク修正、事実の更新 / トリムを指す。
- **承認後に実行（不可逆）**は、ファイル削除、メモリの統合を指す。実行前に「対象・理由・代替の所在」を一覧で提示する。

### 6. 索引同期

- `MEMORY.md` を実態に合わせる。1 メモリ = 1 行、目安 150 字以内（`- [Title](file.md) — hook`）。メモリ本文は索引に書かない。**7 で階層化済みの場合はカテゴリ単位に読み替える**（トップレベル `MEMORY.md` は1カテゴリ=1行、fact 単位の1行ルールは各サブ索引側に適用する。フラット構成へ戻さない）。

### 7. 索引が肥大化した場合の再編（閾値に近づいた時のみ）

- 150 字/行のトリムを終えてもなお `MEMORY.md` が読み込み上限（200 行 / 25KB）に近づいている場合**だけ**、カテゴリ単位の階層化を検討する。余裕がある間に先回りして再編しない（過剰設計）。
- 再編方法: fact ファイルをテーマ別のサブディレクトリへ束ね、各サブディレクトリに短いサブ索引（例: `<category>/INDEX.md`）を置く。トップレベルの `MEMORY.md` は個々の fact でなく**カテゴリ単位で1行**だけを持ち、そのサブ索引を指す。fact 数が増えてもカテゴリ数はずっと緩やかにしか増えないため、`MEMORY.md` 自体の肥大化を先送りできる。
- fact ファイル自体の内容・命名規則・1ファイル=1事実の原則は変えない。索引の階層を一段増やすだけ。

## 出力

1. 検出結果（メモリ別に: 状態 / 判定 / 理由。MCP/外部サイトで未検証だった項目はその旨を明記）。
2. 自動適用した安全な修正の一覧。
3. 承認待ちの不可逆操作（削除・統合）の一覧と推奨。
4. 利用実感の確認が必要な項目（あれば、理由付きで）。
